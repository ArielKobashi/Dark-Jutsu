2. Regra de estado visual

2.1. Icone verde
- Significa: este PC esta respondendo na API 8765 e e o servidor ativo.
- Acao esperada do menu: "Tornar este PC o reserva".
- Efeito esperado: parar a API local para que outro servidor assuma.

2.2. Icone vermelho
- Significa: outro PC esta respondendo na API 8765.
- Acao esperada do menu: "Tornar este PC o Principal".
- Efeito esperado: tentar tornar este PC o servidor ativo.

2.3. Icone preto
- Significa: nenhum servidor respondeu na API 8765.
- Acao esperada do menu: "Tornar este PC o Principal".
- Efeito esperado: tentar iniciar PostgreSQL local e API local.

2.4. Senha dos botoes
- Botoes administrativos pedem senha: 654321.
- "Abrir Dark-Jutsu" nao deve pedir senha.


3. Regra automatica do guardiao

3.1. Frequencia desejada
- Guardiao deve rodar a cada 1 minuto.
- Entrada preferencial: \\fileserver\...\scripts\iniciar_guardiao_servidor_oculto.vbs
- Esse VBS chama: \\fileserver\...\scripts\guardiao_servidor_tick_darkjutsu.bat

3.2. Quando a principal responde
- Se 192.168.5.44 /health responde ok:true:
  - principal permanece ativa;
  - reserva deve parar a API local se ela estiver ligada;
  - contadores de estado preto devem ser limpos.

3.3. Quando a reserva responde
- Se 192.168.5.38 /health responde ok:true e a principal nao responde:
  - reserva continua ativa;
  - se a principal voltar e nao estiver pausada, a principal deve reassumir;
  - apos principal reassumir, reserva deve parar localmente na rodada seguinte.

3.4. Quando nenhum servidor responde, visto pela principal
- A principal deve tentar iniciar imediatamente.
- Fluxo esperado:
  - verificar PostgreSQL local com pg_isready;
  - iniciar PostgreSQL se necessario;
  - iniciar API em 0.0.0.0:8765.

3.5. Quando nenhum servidor responde, visto pela reserva
- 0 a 3 minutos preto:
  - reserva aguarda;
  - nao assume imediatamente.
- Apos 3 minutos preto:
  - reserva tenta acionar a principal inicial remotamente.
  - tentativa remota prevista: schtasks /Run /S 192.168.5.44 /TN "Dark-Jutsu Guardiao Servidor"
- Apos 5 minutos preto:
  - se principal ainda nao respondeu, reserva assume.

3.6. Observacao critica
- A versao atualmente publicada do guardiao no compartilhamento ainda usa PowerShell em varias partes:
  - detectar IP com Get-NetIPAddress;
  - testar /health com Invoke-RestMethod;
  - controlar contador de tempo preto com Get-Date;
  - controlar pausa da principal.
- Isso e incompatibilidade forte com o ambiente atual, pois PowerShell tem retornado "Acesso negado".
- Regra correta: scripts de guardiao, teste, backup e restore devem evitar PowerShell onde possivel.


4. Regra de assumir servidor

4.1. Script
- \\fileserver\...\scripts\assumir_servidor_darkjutsu.bat

4.2. Logica
- Verifica PostgreSQL local com:
  C:\DarkJutsu\PostgreSQL\pgsql\bin\pg_isready.exe -h 127.0.0.1 -p 5433
- Se PostgreSQL nao estiver pronto, chama iniciar_postgres_darkjutsu.bat.
- Se PostgreSQL ficar pronto, inicia a API via:
  iniciar_api_darkjutsu_service.vbs

4.3. Pontos bons
- Nao depende da porta 5433 estar "limpa"; aceita PostgreSQL ja pronto.
- API e iniciada em segundo plano.

4.4. Risco
- iniciar_api_darkjutsu_service.vbs chama api\iniciar_api_servidor.bat local.
- Se a copia local do app estiver desatualizada, a API pode subir versao antiga.
- Recomendacao: pacote local deve ser atualizado antes ou durante configuracao.


5. Regra da API

5.1. Script local
- %USERPROFILE%\Desktop\Dark-Jutsu\api\iniciar_api_servidor.bat

5.2. Script publicado
- \\fileserver\...\scripts\iniciar_api_darkjutsu.bat
- \\fileserver\...\scripts\iniciar_api_darkjutsu_service.vbs

5.3. Comportamento esperado
- Host: 0.0.0.0
- Porta: 8765
- DATABASE_URL padrao:
  postgresql://dark_jutsu:dark_jutsu_dev@127.0.0.1:5433/dark_jutsu

5.4. Problema ja observado
- Python portable pode falhar com:
  "Failed to import encodings module"
- Causa: ambiente PYTHONHOME/PYTHONPATH ausente em algumas formas de chamada.
- O iniciar_api_servidor.bat local foi ajustado para procurar Python via CMD puro, sem PowerShell.


6. Regra do PostgreSQL

6.1. Script
- iniciar_postgres_darkjutsu.bat

6.2. Comportamento esperado
- Se porta 5433 ja esta LISTENING, considera PostgreSQL ativo e sai com sucesso.
- Se nao esta ativo, chama pg_ctl start.
- Log principal: C:\DarkJutsu\logs\postgres_startup.log
- Log runtime: C:\DarkJutsu\logs\postgres_runtime.log

6.3. Risco
- O script ainda usa PowerShell apenas para sleep de 5 segundos.
- Isso pode ser substituido por:
  timeout /t 5 /nobreak >nul


7. Regra de parar API

7.1. Script
- parar_api_darkjutsu.bat

7.2. Comportamento esperado
- Localiza processo ouvindo na porta 8765 via netstat.
- Mata o PID com taskkill /F.
- Nao para PostgreSQL.

7.3. Observacao
- "Encerrar servidor local" atualmente para a API, nao o PostgreSQL.
- Isto e coerente com o desenho: servidor ativo e definido pela API 8765.


8. Regra de backup e restore

8.1. Backup
- Script: backup_postgres_darkjutsu.bat
- A maquina gera backup se a API local responder em http://127.0.0.1:8765/health.
- Backup vai para:
  \\fileserver\...\backups\darkjutsu_backup_yyyyMMdd_HHmmss.backup
- Mantem os ultimos 72 backups.

8.2. Restore
- Script: restaurar_backup_reserva_darkjutsu.bat
- Se API local esta ativa na 8765, restore e ignorado.
- Se API local esta inativa, restaura backup mais recente.
- Controla ultimo backup restaurado em:
  C:\DarkJutsu\logs\ultimo_backup_restaurado.txt

8.3. Risco critico
- Backup e restore ainda dependem de PowerShell para:
  - gerar timestamp;
  - testar API local;
  - localizar backup mais recente;
  - limpar backups antigos.
- Com PowerShell bloqueado, sincronizacao pode falhar silenciosamente.

8.4. Risco de arquitetura
- Isto nao e replicacao simultanea real.
- E modelo ativo/passivo por backup e restore a cada 5 minutos.
- Perda possivel: alteracoes feitas depois do ultimo backup valido podem nao estar no outro PC.

9.4. Risco observado
- Versao publicada no compartilhamento pode estar diferente da local.
- O .bat publicado ainda pode estar sem PYTHONHOME/PYTHONPATH.

  iniciar_monitor_reserva_python_darkjutsu.bat

12. Inconsistencias encontradas

12.1. CRITICO - PowerShell bloqueado
- O ambiente mostrou repetidamente "Acesso negado" ao rodar PowerShell.
- Mesmo assim scripts importantes ainda dependem dele:
  - guardiao publicado;
  - testar-servidor;
  - backup;
  - restore;
  - instaladores de atalho;
  - monitor antigo.
- Acao recomendada: migrar scripts operacionais para CMD puro ou Python.

12.2. CRITICO - guardiao local ausente
- Na pasta local C:\Users\Davi.souza\Desktop\Dark-Jutsu\scripts nao aparece:
  guardiao_servidor_tick_darkjutsu.bat
- Mas o VBS local chama a versao publicada na rede.
- Se a rede estiver certa, funciona; se precisar editar local, falta arquivo base.
- Acao recomendada: recriar localmente a versao oficial e publicar.

12.3. ALTO - guardiao publicado ainda usa PowerShell
- Mesmo apos tentativa de criar versao CMD puro, o arquivo publicado ainda contem PowerShell.
- Acao recomendada: substituir a versao publicada por uma versao CMD/Python puro.

12.4. ALTO - pausa da principal depende de PowerShell
- tornar_reserva_operacional_darkjutsu.bat e tornar_principal_operacional_darkjutsu.bat criam principal-pausado-ate.txt usando PowerShell.
- Se PowerShell falha, o arquivo de pausa nao e criado, mas o log diz que criou.
- Acao recomendada: trocar por contador simples ou arquivo sentinela sem data.

12.5. ALTO - monitor PowerShell obsoleto ainda instalado em alguns caminhos
- Instaladores ainda apontam para iniciar_monitor_principal_powershell_oculto.vbs, que chama PowerShell.
- Acao recomendada: instaladores devem apontar para iniciar_monitor_reserva_python_darkjutsu.bat.

12.6. MEDIO - comandos com acentos quebrados
- Alguns caminhos aparecem como "aplica????????es code" em BAT.
- Isso pode quebrar em ambientes com codepage diferente.
- Acao recomendada: preferir busca wildcard "aplica* code" ou caminho sem acento.

12.7. MEDIO - backup/restore nao sao replicacao em tempo real
- O sistema fica "quase atualizado", mas nao simultaneo.
- Acao recomendada: manter expectativa como ativo/passivo por backup, nao HA real.

12.8. MEDIO - schtasks remoto pode falhar
- A reserva tenta acionar principal por schtasks /S 192.168.5.44.
- Isso exige permissao remota, firewall e politica do Windows.
- Se falhar, a regra ainda assume reserva apos 5 minutos.


13. Logica desejada final, simplificada

13.1. Fonte de verdade
- Servidor ativo e quem responde /health na porta 8765.
- PostgreSQL sozinho na 5433 nao significa sistema ativo.

13.2. Principal
- Se principal esta viva, todos usam principal.
- Reserva nao deve ficar com API ativa.

13.3. Reserva
- Se principal caiu e reserva ficou vendo preto:
  - aguarda 3 minutos;
  - tenta chamar principal;
  - se em mais 2 minutos nao voltou, assume.

13.4. Retorno da principal
- Quando principal voltar, ela reassume.
- Reserva para API local na proxima rodada.

13.5. Usuario
- Abre app pelo index.html do compartilhamento.
- Monitor mostra estado:
  - verde: este PC ativo;
  - vermelho: outro PC ativo;
  - preto: nenhum ativo.
