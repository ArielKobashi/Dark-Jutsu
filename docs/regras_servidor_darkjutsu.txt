Dark-Jutsu - regras e descricao dos servidores, APIs, guardiao e monitor
Revisao: 2026-07-11

1. Objetivo

O Dark-Jutsu usa dois PCs fixos como servidores locais em modo ativo/passivo:

- Principal: deve ser o servidor preferencial sempre que estiver disponivel.
- Reserva: assume automaticamente quando nenhum servidor responde.

O servidor ativo e definido pela API respondendo em /health na porta 8765.
PostgreSQL sozinho na porta 5433 nao significa que o sistema esta ativo.

5. Instalador unico

5.1. Comando oficial

Rodar uma vez em cada usuario do Windows que precisa ter o icone e o guardiao:

cmd /c "pushd \\fileserver\Almoxarifado\0800\servidor\dark-jutsu\scripts && call instalar_atualizar_guardiao_monitor_darkjutsu.bat && popd"

5.2. Decisao por IP

O instalador detecta o IP local:

- Se encontrar 192.168.5.44:
  - papel = PRINCIPAL;
  - monitor = POWERSHELL;
  - inicia monitor_principal_powershell_darkjutsu.ps1.

- Se encontrar 192.168.5.38:
  - papel = RESERVA;
  - monitor = PYTHON;
  - copia monitor Python para %LOCALAPPDATA%\DarkJutsu\monitor;
  - garante Python portable no usuario, copiando da rede se faltar;
  - inicia monitor_reserva_python_darkjutsu.py.

- Se nao encontrar nenhum dos dois IPs:
  - nao instala;
  - registra falha;
  - evita instalar versao errada em outro PC.

5.3. O que o instalador faz

- Verifica acesso ao servidor de arquivos.
- Verifica arquivos obrigatorios.
- Cria pasta local do monitor.
- No reserva, garante Python portable.
- Garante copia estavel da API em C:\DarkJutsu\Dark-Jutsu\api.
- Cria guardiao local em loop escondido.
- Configura inicializacao automatica com fallback.
- Reinicia o guardiao.
- Reinicia o monitor correto para o IP da maquina.
- Executa checks finais.
- Exibe RESULTADO: OK ou RESULTADO: FALHOU.

5.4. Log do instalador

- Log local:
  C:\DarkJutsu\logs\instalador_guardiao_monitor.log


6. Guardiao automatico

6.1. Arquivos

- Script principal:
  scripts\guardiao_servidor_tick_darkjutsu.bat
- Inicializador oculto:
  scripts\iniciar_guardiao_servidor_oculto.vbs
- Alias de compatibilidade:
  scripts\guardiao_continuo_tick_darkjutsu.bat
  scripts\guardiao_continuo_tick_darkjutsu_hidden.vbs

6.2. Frequencia

- O instalador cria um loop local em:
  %LOCALAPPDATA%\DarkJutsu\monitor\guardiao_loop_darkjutsu.vbs
- Esse loop chama o guardiao a cada 60 segundos.
- Portanto, uma queda deve ser tratada em ate aproximadamente 1 minuto.

6.3. Regra quando a principal responde

Se http://192.168.5.44:8765/health responde ok:true:

- principal e considerada ativa;
- reserva deve parar a API local se estiver ligada;
- contadores de estado preto sao limpos;
- nenhuma assuncao e feita.

6.4. Regra quando a reserva responde

Se http://192.168.5.38:8765/health responde ok:true e a principal nao responde:

- reserva continua ativa;
- principal, quando voltar, deve reassumir;
- depois da principal responder, a reserva para a API local na rodada seguinte.

6.5. Regra quando nenhum servidor responde

Se principal e reserva estao OFF:

- visto pela principal:
  - principal tenta assumir imediatamente.

- visto pela reserva:
  - reserva tenta assumir imediatamente no proximo tick;
  - nao espera mais 3 ou 5 minutos.

6.6. Comando manual para rodar um tick do guardiao

cmd /c "pushd \\fileserver\Almoxarifado\0800\servidor\dark-jutsu\scripts && call guardiao_servidor_tick_darkjutsu.bat && popd"


7. Assumir servidor

7.1. Script

- scripts\assumir_servidor_darkjutsu.bat

7.2. Fluxo

- Detecta IP local.
- Registra tentativa no log compartilhado.
- Testa API local em 127.0.0.1:8765.
- Se API ja responde, termina OK.
- Testa PostgreSQL com pg_isready.
- Se PostgreSQL nao responde, chama iniciar_postgres_darkjutsu.bat.
- Se PostgreSQL continuar falhando, registra ERRO e termina com falha.
- Solicita inicio da API por iniciar_api_darkjutsu_service.vbs.
- Aguarda ate 30 segundos testando /health.
- So retorna OK se a API responder.
- Se a API nao responder, registra ERRO.

7.3. Importante

Antes, assumir servidor apenas solicitava API em segundo plano.
Agora ele valida se a API realmente subiu.


8. API

8.1. Arquivos

- API Python:

- Iniciador local da API:
  api\iniciar_api_servidor.bat
- Inicializador oculto/servico:
  scripts\iniciar_api_darkjutsu_service.vbs

8.2. Raiz usada para iniciar

O iniciar_api_darkjutsu_service.vbs procura nesta ordem:

1. %USERPROFILE%\Desktop\Dark-Jutsu
2. C:\DarkJutsu\Dark-Jutsu

Se nao encontrar api\iniciar_api_servidor.bat em nenhum dos dois, registra erro.

8.3. Motivo da copia em C:\DarkJutsu\Dark-Jutsu

Usuarios diferentes da mesma maquina podem nao ter Desktop\Dark-Jutsu.
Por isso o instalador cria uma copia estavel da API em C:\DarkJutsu\Dark-Jutsu\api.

8.4. Variaveis padrao

- DARK_JUTSU_API_HOST=0.0.0.0
- DARK_JUTSU_API_PORT=8765
- DATABASE_URL=postgresql://dark_jutsu:dark_jutsu_dev@127.0.0.1:5433/dark_jutsu

10. Monitor visual

10.1. Cores

- Verde:
  - este PC e o servidor ativo;
  - API local responde na porta 8765.

- Vermelho:
  - outro PC e o servidor ativo;
  - este PC nao deve servir API.

- Preto:
  - nenhum servidor respondeu.

10.2. Menu esperado

- Abrir Dark-Jutsu:
  - nao pede senha;
  - abre o index.html do compartilhamento.

- Testar servidor:
  - pede senha 654321;
  - abre CMD visivel.

- Verificar/iniciar agora:
  - pede senha 654321;
  - roda guardiao_servidor_tick_darkjutsu.bat;
  - no monitor Python da reserva, abre CMD visivel com retorno.

- Tornar este PC o Principal:
  - aparece quando este PC nao esta ativo;
  - pede senha 654321;
  - tenta assumir servidor.

- Tornar este PC o reserva:
  - aparece quando este PC esta ativo;
  - pede senha 654321;
  - para API local.

- Encerrar servidor local:
  - pede senha 654321;
  - para API local.

10.3. Diferenca proposital entre principal e reserva

- Principal usa PowerShell porque ali esta funcionando.
- Reserva usa Python porque PowerShell apresentou "Acesso negado" no ambiente do Jean.
- Essa diferenca e proposital e controlada por IP.

15. Riscos conhecidos

15.1. Firewall ou TI

Outros PCs so acessam a API se a porta 8765 estiver liberada na rede privada.
Se o firewall bloquear, /health nao responde pela rede.

15.2. Permissao de usuario

Alguns usuarios bloqueiam:

- Registro HKCU Run;
- pasta Inicializar;
- tarefa agendada;
- PowerShell.

O instalador tenta fallbacks, mas se todos forem bloqueados a inicializacao automatica pode falhar.

15.3. PowerShell

PowerShell e usado no monitor principal porque funciona no principal.
No reserva, a regra oficial e Python.

15.4. Backup nao e replicacao real

O sistema nao tem replicacao PostgreSQL nativa.
E importante manter backups frequentes e entender que pode haver perda entre o ultimo backup e a queda.

15.5. Usuario novo

Usuario novo precisa rodar o instalador unico pelo menos uma vez para ter icone/guardiao no login.
No reserva, o instalador copia Python portable se faltar.


16. Fluxo esperado em queda

16.1. Principal desligou

1. Reserva fica preta.
2. Guardiao da reserva roda em ate 60 segundos.
3. Guardiao ve principal OFF e reserva OFF.
4. Reserva chama assumir_servidor_darkjutsu.bat.
5. PostgreSQL local e iniciado se necessario.
6. API local e iniciada.
7. /health e testado por ate 30 segundos.
8. Icone da reserva fica verde.
9. Log compartilhado registra cada etapa.

16.2. Principal voltou

1. Guardiao da principal roda.
2. Principal ve reserva ativa e principal livre.
3. Principal reassume.
4. Reserva ve principal ativa na rodada seguinte.
5. Reserva para API local.
6. Icones ficam:
   - principal verde;
   - reserva vermelho.
